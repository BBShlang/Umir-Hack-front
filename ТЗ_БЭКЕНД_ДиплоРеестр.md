# ТЕХНИЧЕСКОЕ ЗАДАНИЕ — Бэкенд «ДиплоРеестр» (Java)

## 1. Общее описание

Платформа **«ДиплоРеестр»** — веб-приложение для верификации дипломов образовательных учреждений. Система позволяет:

- **ВУЗам** — загружать реестры дипломов (CSV/Excel), управлять записями (поиск, фильтрация, отзыв), настраивать ЭЦП, QR-коды, API-ключи
- **Студентам** — просматривать свои дипломы, генерировать QR-коды для шаринга с работодателями, отслеживать историю проверок
- **HR (работодателям)** — проверять подлинность дипломов по серийному номеру, хешу SHA-256 или QR-коду; проводить массовую верификацию; получать отчёты

### 4.3. Diploma (Реестр дипломов)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | `BIGINT (PK, auto)` | Идентификатор |
| `serial_number` | `VARCHAR(30) NOT NULL` | Серийный номер (уникальный в рамках ВУЗа) |
| `university_id` | `BIGINT (FK → User) NOT NULL` | ВУЗ-владелец |
| `student_name` | `VARCHAR(255) NOT NULL` | ФИО выпускника |
| `specialty` | `VARCHAR(255) NOT NULL` | Специальность |
| `degree` | `VARCHAR(50) NOT NULL` | Степень (Бакалавр/Магистр/Специалист) |
| `issue_date` | `DATE NOT NULL` | Дата выдачи |
| `year` | `VARCHAR(4) NOT NULL` | Год выпуска |
| `status` | `ENUM('active','revoked','pending') NOT NULL DEFAULT 'active'` | Статус |

| `revoked_by` | `BIGINT (FK → User)` | Кто отозвал |

**Индексы:**
- `UNIQUE(university_id, serial_number)` — уникальность серийного номера в рамках ВУЗа
- `INDEX(hash)` — быстрый поиск по хешу
- `INDEX(student_name)` — поиск по ФИО
- `INDEX(status)` — фильтрация по статусу

## 5. API Endpoints

Все эндпоинты возвращают JSON. Формат ошибки:
```json
{ "detail": "Текст ошибки" }
```
HTTP-статус ошибки передаётся в поле `err.status` на клиенте.

### 5.1. Аутентификация

#### `POST /api/v1/auth/login`
**Body:**
```json
{ "username": "email@example.com", "password": "secret" }
```
**Response 200:**
```json
{
  "access_token": "jwt...",
  "refresh_token": "refresh...",
  "user": { "id": 1, "name": "МГТУ", "role": "university" }
}
```
**Response 401:** `{ "detail": "Неверный email или пароль" }`

#### `POST /api/v1/auth/refresh`
**Body:**
```json
{ "refresh_token": "refresh..." }
```
**Response 200:**
```json
{ "access_token": "new_jwt...", "refresh_token": "new_refresh..." }
```

#### `POST /api/v1/auth/logout`
**Auth:** Bearer token
**Body:**

### 5.2. Дипломы (CRUD)

#### `GET /api/v1/diplomas`
**Auth:** Bearer token
**Query params:**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `page` | int | Номер страницы (по умолч. 1) |
| `pageSize` | int | Размер страницы (по умолч. 10) |
| `search` | string | Поиск по serialNumber, studentName, specialty |
| `status` | string | Фильтр: `all`, `active`, `revoked`, `pending` |
| `sort` | string | Сортировка: `serialNumber`, `studentName`, `issueDate` |
| `sortDir` | string | Направление: `asc`, `desc` |

**Response 200:**
```json
{
  "data": [
    {
      "id": 1,
      "serialNumber": "DIP-2023-001",
      "studentName": "Иванов Иван Иванович",
      "specialty": "Информатика и ВТ",
      "degree": "Бакалавр",
      "university": "МГТУ им. Баумана",
      "issueDate": "2023-06-20",
      "status": "active",
      "hash": "a1b2c3...64hex",
      "year": "2023",
      "createdAt": "2023-06-20T10:00:00Z"
    }
  ],
  "total": 1250,
  "page": 1,
  "pageSize": 10
}
```

**Access control:**
- `university` — видит только свои дипломы
- `student` — видит только свои дипломы (по email)
- `hr` — **не имеет доступа** (HR использует verify-эндпоинты)

#### `GET /api/v1/diplomas/{id}`
**Auth:** Bearer token
**Response 200:** объект Diploma (как выше)

#### `POST /api/v1/diplomas`
**Auth:** Bearer token (только `university`)
**Body:**
```json
{
  "serialNumber": "DIP-2026-00001",
  "studentName": "Петров Пётр Петрович",
  "specialty": "Программная инженерия",
  "degree": "Бакалавр",
  "issueDate": "2026-06-30"
}
```
Сервер автоматически:
- Вычисляет `hash = SHA-256(serialNumber + studentName + specialty + issueDate)`
- Устанавливает `year` из `issueDate`
- Устанавливает `status = 'active'`
- Привязывает к `university_id` из токена

**Response 201:** созданный объект Diploma

#### `PATCH /api/v1/diplomas/{id}`
**Auth:** Bearer token (только `university`, только свои)
**Body:** частичное обновление допустимых полей
**Response 200:** обновлённый объект

#### `DELETE /api/v1/diplomas/{id}` → **Отзыв**
**Auth:** Bearer token (только `university`, только свои)
Фактически: `status = 'revoked'`, `revoked_at = NOW()`, `revoked_by = user_id`
**Response 204**

### 5.3. Верификация дипломов

#### `GET /api/v1/verify`
**Auth:** Bearer token (HR) ИЛИ API-ключ в заголовке `Authorization: Bearer sk-prod-...`
**Query params (один из):**
| Параметр | Тип | Описание |
|----------|-----|----------|
| `serial` | string | Серийный номер диплома |
| `hash` | string | SHA-256 хеш |

**Response 200 (найден, активен):**
```json
{
  "valid": true,
  "status": "active",
  "serialNumber": "DIP-2023-001",
  "university": "МГТУ им. Баумана",
  "specialty": "Информатика и ВТ",
  "degree": "Бакалавр",
  "issueDate": "2023-06-20",
  "hash": "a1b2c3..."
}
```

**Response 200 (отозван):**
```json
{
  "valid": false,
  "status": "revoked",
  "serialNumber": "DIP-2021-555",
  "university": "МГУ им. Ломоносова",
  "reason": "Диплом отозван ВУЗом"
}
```

**Response 200 (не найден):**
```json
{
  "valid": false,
  "status": "not_found",
  "reason": "Диплом не найден в реестре"
}
```

**Rate limiting:** 1000 запросов/день на API-ключ. При превышении — `429 Too Many Requests`.

#### `POST /api/v1/verify/bulk`
**Auth:** Bearer token (HR) ИЛИ API-ключ
**Body:**
```json
{ "serials": ["DIP-001", "DIP-002", "DIP-003"] }
```
**Response 200:**
```json
{
  "total": 3,
  "valid": 2,
  "invalid": 1,
  "results": [ /* массив результатов как в GET /verify */ ]
}
```

#### `GET /api/v1/verify/share/{token}`
**Без авторизации** (публичный эндпоинт)
**Response 200:**
```json
{
  "valid": true,
  "status": "active",
  "diploma": { /* данные диплома */ },
  "expiresAt": "2026-04-11T10:00:00Z"
}
```
**Response 410 (истёк):**
```json
{ "valid": false, "status": "expired", "reason": "Срок действия ссылки истёк" }
```

## 6. Бизнес-правила

### 6.1. Генерация хеша
```
SHA-256(serial_number + "|" + student_name + "|" + specialty + "|" + issue_date)
```
Хеш должен быть уникальным в рамках всей таблицы дипломов.

### 6.2. Rate Limiting
- **API-ключ HR:** 1000 запросов/день (сброс в 00:00 UTC)
- **Верификация через UI:** 10 запросов/минуту (на IP + пользователя)
- При превышении: HTTP `429` с телом `{ "detail": "Превышен лимит запросов. Подождите минуту." }`

### 6.3. Валидация входных данных
| Поле | Правило |
|------|---------|
| Email | `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` |
| Пароль | минимум 6 символов |
| Серийный номер | `/^[A-Z0-9а-яА-ЯёЁ\-]{6,30}$/i` |
| SHA-256 хеш | `/^[a-fA-F0-9]{64}$/` |
| Файл реестра | CSV, XLSX; максимум 10 МБ |
| TTL QR-ссылки | 1–30 дней |
| Макс. записей в bulk | 1000 |

### 6.4. Отзыв диплома
- Только ВУЗ-владелец может отозвать свой диплом
- При отзыве: `status = 'revoked'`, `revoked_at = NOW()`
- Все активные share-токены этого диплома должны быть деактивированы
- В журнал верификаций пишется запись при каждой проверке отозванного диплома

### 6.5. CSV/Excel парсинг
- Автоматическое определение разделителя: `,` или `;`
- Кодировка: UTF-8
- Ожидаемые заголовки: `ФИО`, `Серийный номер`, `Специальность`, `Дата выдачи`
- Дата: формат `YYYY-MM-DD` или `DD.MM.YYYY`
- Максимум 1000 записей за раз
- Дубликаты серийных номеров (в рамках одного ВУЗа) пропускаются с ошибкой

## 7. Безопасность

### 7.1. JWT
- **Access token:** JWT, время жизни 15 минут
- **Refresh token:** opaque string, время жизни 7 дней
- Хранить refresh tokens в БД с возможностью отзыва
- Payload access token: `{ sub: userId, role, name, exp, iat }`

### 7.2. CORS
Разрешить CORS для фронтенд-домена (указан в `.env` как `VITE_API_BASE_URL`).

### 7.3. API-ключи
- Хранить только хеш ключа (SHA-256)
- При создании показывать полный ключ **однократно**
- Префикс (`sk-prod-XXXXXX`) хранить в открытом виде для идентификации

### 7.4. Rate limiting
Реализовать через Redis (sliding window или token bucket).

### 7.5. SQL Injection
Использовать параметризованные запросы (JPA Criteria / jOOQ / MyBatis).

## 11. Примечания для бэкенд-разработчика

1. **Фронтенд уже готов** и ожидает строго указанные пути и форматы ответов. Не меняйте контракт без согласования.
2. **BASE_URL** фронтенда: все запросы идут на `${BASE_URL}/api/v1/...`. В dev-режиме Vite проксирует `/api` на бэкенд.
3. **Авторизация** — Bearer token в заголовве `Authorization`. Фронтенд берёт токен из `localStorage.getItem('auth_token')`.
4. **Mock-данные** на фронтенде используются только для демонстрации. Бэкенд должен заменить все `setTimeout`-заглушки реальными данными.
5. **ShareView.vue** — файл отсутствует на фронте (маршрут `/student/share/:id?` есть в роутере). Это проблема фронтенда, бэкенд должен реализовать `/api/v1/share-tokens` независимо.
6. **Маршрут `forbidden`** — referenced в `useRoleGuard.js` но не определён в роутере. Это проблема фронтенда.
7. Фронтенд использует `api.js` с готовыми методами — сверяйтесь с ним при реализации контрактов.
8. Файл `api-mock.js` содержит эталонную mock-логику — используйте её как референс поведения.
